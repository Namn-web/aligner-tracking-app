# 設計書 — マウスピース管理アプリ（kyousei）

他のAIモデル・開発者への引き継ぎ用ドキュメントです。
コードを変更する前に、このファイルと `BRAND.md`（デザインシステム）を必ず読んでください。

---

## 1. アプリ概要

| 項目 | 内容 |
|---|---|
| アプリ名 | マウスピース管理 |
| 用途 | インビザライン（歯科矯正マウスピース）の装着時間・交換ステージ管理 |
| ターゲット | 矯正治療中の個人ユーザー（スマートフォンで利用） |
| プラットフォーム | PWA（インストール可能なWebアプリ） |
| 状態 | 開発中。**リリース準備フェーズ**（2026-07時点） |

### ユーザーの基本フロー

1. 初回起動時「今、装着していますか？」に回答（セットアップ）
2. 以降、マウスピースの着脱のたびに中央の円形トグルボタンを押す
3. アプリが「今日の装着時間」を計算し、目標（既定22時間）との差を表示
4. 目標達成日が交換周期（7〜10日）に達したら「交換した」を押して次の1枚へ

## 2. 技術構成

- **バニラ HTML / CSS / JS**。フレームワーク・ビルドツール・npm依存なし
- 外部依存は Google Fonts（M PLUS Rounded 1c）のみ
- データ保存は **localStorage のみ**（サーバーなし・アカウントなし）
- 時刻は全て **エポックミリ秒（`Date.now()`）** で統一

## 3. データ設計（localStorage）

キーはすべて `kyousei_` プレフィックス付き。`app.js` 冒頭の `STORAGE_KEYS` に定義があります。

| キー | 型 | 既定値 | 内容 |
|---|---|---|---|
| `kyousei_events` | `[{type:'on'\|'off', ts:number}]` | `[]` | **着脱イベントログ（最重要データ）**。時系列の配列 |
| `kyousei_stages` | `[{start:number, end:number\|null}]` | `[]` | マウスピース各枚の期間。最後の要素が現在の1枚（`end:null`） |
| `kyousei_dayOverrides` | `{'YYYY-MM-DD': number}` | `{}` | 履歴画面からの手動修正値（その日の装着ms）。**ログより優先** |
| `kyousei_goalHours` | number | `22` | 1日の目標装着時間 |
| `kyousei_replaceCycle` | number | `7` | 交換周期（7/8/9/10のみ有効。他の値は7にフォールバック） |
| `kyousei_totalAligners` | number \| なし | `null` | 合計枚数（任意設定） |
| `kyousei_alertMinutes` | number | `60` | 外し時間アラート閾値（0で無効） |
| `kyousei_undoStack` | スナップショット配列 | `[]` | 元に戻す用。最大20件。events/stages/goalHours/totalAligners/overridesの深いコピー |

### イベントログ方式（中核の考え方）

装着時間は「合計値」を保存するのではなく、**着脱イベントの時刻列から毎回計算**します。

- `wornMsInRange(events, start, end)` — 期間内で状態が `'on'` だった合計msを算出。期間開始前の最後のイベントから初期状態を決める
- `getDayWornMs(dayStart, now)` — その日の装着ms。**`overrides` にキーがあればそれを優先**、なければログから計算
- 状態変更は必ず `pushUndo()` → データ変更 → `save*()` → `render()` の順で行うこと

## 4. 主要ロジック（app.js）

### レンダリング

- `render()` が唯一の画面更新入口。**30秒ごとに `setInterval` で自動再実行**される
- 内部で `renderStage` / `renderChecklist` / `renderHistory` / `checkWearAlert` を呼ぶ
- セットアップ済みか（`events.length > 0`）で `#setup` と `#mainView` を出し分け

### ステージ（交換周期）の判定

- `fullDayStarts(stageStart, now)` — ステージ開始後に「丸1日」経過した日の0時一覧。**開始当日は数えず、翌日0時から起算**
- `stageProgress(stage, now)` — 目標達成日数 `met` を数え、`met >= replaceCycle` で交換可能（`ready`）。未達成日があるとその分交換が延びる設計
- `getStageCumulativeDeficit` — ステージ内の累計不足時間
- 交換目安日 `estimatedTs` = 今日 + 残り必要達成日数

### その他の仕様

- **装着時の確認**：外→装着に切り替える際、`confirm('歯磨きをしましたか？…')` を挟む（仕様。削除しないこと）
- **外し時間アラート**：`checkWearAlert` が外し続き時間 ≥ `alertMinutes` で画面内バナー＋Web通知。クールタイムは `alertMinutes` と同じ（`lastAlertTs` はメモリのみ）
- **履歴**：既定は直近10日。期間指定（date input×2）で最大180日まで遡れる。各行の時間は編集可能で `overrides` に保存
- **リセット**：`resetAll()` は confirm 後に全データ初期化（undoスタックには積む）

## 5. UI構成

シングルHTML内に3画面。ページ遷移はなく、CSSの `transform` でスライドイン。

| 画面 | 要素ID | 出し方 |
|---|---|---|
| メイン | `#mainView` | 常時表示（セットアップ後） |
| 設定 | `#settingsPage` | ハンバーガーメニュー → `.open` クラスで右からスライド |
| Q&A | `#qaPage` | 同上。アコーディオン形式 |

メイン画面の構成（上から）：
1. 外し時間アラートバナー（`#wearAlert`、通常非表示）
2. **円形トグルボタン＋進捗リング**（アプリの見せ場。SVGリング＋クレイボタン）
3. 今日の装着時間カード（ヒーロー、白×ティールグラデ）
4. ステージカード（交換管理。ティールのトップボーダー）
5. チェックリスト（フラット表示、カードなし）
6. 履歴カード（インセット＝凹んだ表現）

※ カードの視覚階層（4段階）は意図的な設計。全カードを同じ見た目に戻さないこと。詳細は `BRAND.md`。

## 6. デザイン原則（要点のみ、詳細は BRAND.md）

- **クレイモーフィズム**：粘土のような凹凸をbox-shadowの多層指定で表現。これがアプリの核となる個性。**維持すること**
- フォントは **M PLUS Rounded 1c**（丸ゴシック）。ウェイトは500/700/900のみ
- **絵文字は使わない**。アイコンはインラインSVGで統一（歯・ケース・歯車・吹き出し）
- シャドウは純黒でなくウォームブラウン `rgba(63,55,43,…)`
- ボタンは `:active` で凸→凹シャドウ＋沈み込み（`--clay-pressed` 変数）
- `prefers-reduced-motion` 対応済み。アニメーション追加時はこのメディアクエリを壊さない

## 7. 既知の制約・注意点

1. **Service Workerは意図的に無効化中**（`app.js` 末尾）。過去バージョンのSWが古い画面をキャッシュし続けた問題への対処として、起動時に全SW登録解除＋キャッシュ削除を実行している。**オフライン対応を実装する際は、この解除コードの扱いを含めてキャッシュ戦略を再設計すること**
2. **データはlocalStorageのみ** — ブラウザデータ削除・機種変更で全損する。バックアップ機能が未実装（ロードマップ最優先）
3. **通知はページが開いている間のみ**動作する（Pushサーバーなし）
4. `lastAlertTs` はメモリ保持のみ。リロードするとアラートのクールタイムがリセットされる
5. 期間指定の履歴表示は180日が上限（パフォーマンス対策）
6. iOSのPWAではlocalStorageが長期未使用時に消去される可能性がある（Safariの7日ルール）。リリース時に要検証

## 9. 作業ルール（引き継ぎ側へ）

- フレームワークやビルドツールを**導入しない**（バニラ構成を維持）
- 新しい設定値を追加するときは `STORAGE_KEYS` に追加し、`load*/save*` 関数ペアの既存パターンに従う
- データを変更する操作には必ず `pushUndo()` を先に呼ぶ
- UI文言は日本語。断定調・煽り表現なし（ユーザーのライティングルールに準拠）
- コミットメッセージは日本語（既存の履歴に合わせる）
